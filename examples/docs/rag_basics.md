# RAG 基础

RAG 是 Retrieval-Augmented Generation 的缩写，通常翻译为检索增强生成。它的核心思想是：不要只依赖大模型参数中的知识，而是在回答问题前先从外部知识库中检索相关内容，再把检索结果作为上下文交给大模型生成答案。

一个典型 RAG 系统包含两条链路：离线建库链路和在线问答链路。

离线建库链路负责处理文档。系统会把 PDF、Markdown、网页文本等原始资料解析成纯文本，再按一定规则切成多个 chunk。每个 chunk 会被 embedding 模型转换成向量，并写入向量数据库。为了后续定位答案来源，chunk 通常需要保留标题、文档名、段落位置等元数据。

在线问答链路从用户问题开始。系统先把问题转换成向量，再到向量数据库中检索语义相似的候选片段，通常取 top-k 个结果。随后可以使用 rerank 模型对候选片段重新排序，筛出更相关的上下文。最后，系统把问题、检索上下文和回答约束组合成 prompt，交给大模型生成答案。

chunk size 是 RAG 效果的重要参数。chunk 太大时，一个片段可能包含过多无关信息，导致噪声增加，占用上下文窗口，也可能让模型难以定位关键事实。chunk 太小时，语义容易被切碎，单个片段缺少必要上下文，导致检索结果虽然相似但无法支撑完整回答。实际项目中通常需要结合文档结构和评估结果调整 chunk 策略。

embedding 的作用是把文本映射到向量空间，让语义相近的文本在向量空间中距离更近。向量检索并不是关键词匹配，它更关注语义相似度。用户问题和文档片段都可以被转换成向量，然后通过相似度计算找到候选片段。

top-k 表示向量检索返回多少个候选片段。top-k 较小时，系统延迟更低，上下文更干净，但可能漏掉正确文档。top-k 较大时，召回率可能提高，但也会引入更多噪声，增加模型上下文压力和响应延迟。因此 top-k 需要通过评估而不是凭感觉设置。

rerank 的作用是在向量检索之后重新排序候选片段。向量检索负责快速召回，rerank 更关注问题和候选片段之间的精细相关性。对于候选片段较多、相似文本较多、专业文档较复杂的场景，rerank 往往能提高最终上下文质量。

RAG 可以减少幻觉，但不能完全消除幻觉。常见原因包括：检索没有召回正确文档，召回片段缺少关键上下文，prompt 没有限制模型必须基于引用回答，或者模型在上下文不足时自行补全事实。因此 RAG 系统需要记录引用来源，并分析回答是否真的被检索内容支撑。

评估 RAG 不能只看答案是否流畅。一个回答可能语言自然，但没有引用正确文档，或者覆盖了错误事实。更可靠的评估应包含多个维度：检索是否命中标准文档，回答是否覆盖关键概念，引用是否存在，响应延迟是否可接受，以及失败 case 属于检索失败、引用缺失、答非所问还是模型幻觉。

RAGBench-CN 的第一版评测思路是用问题集驱动测试。每个问题给出期望关键词和标准文档，脚本调用 RAG 系统问答接口后，统计关键词召回率、引用命中率、平均延迟和失败类型。这个方法不完美，但足够作为轻量评测起点，也适合比较 top-k、chunk size、rerank 等参数变化带来的影响。

一个可面试的 RAG 项目需要留下清晰证据：运行命令、环境配置、测试问题集、评估报告、失败 case、源码阅读笔记和参数对比结果。面试时要能分清平台已有能力和自己的工作：RAGFlow 等平台负责知识库和问答链路，评测工具负责问题集、指标、报告和失败分析。
